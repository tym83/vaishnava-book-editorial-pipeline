# Glossary Spec v1

## 1. Назначение

Глоссарий нужен для:

- санскритских терминов;
- индийских имен;
- географических названий;
- названий писаний;
- эпитетов;
- обращений;
- других духовных и культурных единиц, встречающихся вне шлок и внутри прозы.

Он также может хранить:

- формы, встречающиеся в шлоках;
- курсивность;
- склонение;
- конфликтующие варианты;
- решение редактора.

---

## 2. Роль глоссария

Глоссарий — это не просто список слов.  
Это **редакционная база решений**.

Он должен отвечать на вопросы:

- как писать;
- как склонять;
- выделять ли курсивом;
- какие варианты встречались;
- что одобрено;
- на какие прецеденты опираться.

---

## 3. Формат первой версии

Для первой версии рекомендую:

- основная рабочая база: `CSV`
- при необходимости экспорт в `JSON`

Почему:
- прозрачно;
- просто править вручную;
- легко читать скриптами;
- легко открывать в Excel/LibreOffice.

---

## 5. Значение полей

## `id`

Уникальный идентификатор записи.

## `lemma_ru`

Базовая русская форма.

## `lemma_en`

Базовая английская форма или IAST/транслитерация, если есть.

## `category`

Одна из категорий:

- `personal_name`
- `place_name`
- `scripture_title`
- `philosophical_term`
- `institutional_term`
- `honorific`
- `epithet`
- `other`

## `approved_form`

Утвержденное написание для использования в текстах.

## `declension_notes`

Краткие заметки о склонении.

## `italic_required`

`yes / no / conditional`

## `diacritics_policy`

Например:

- `no_diacritics_in_prose`
- `diacritics_in_quotes_only`
- `conditional_inline_quote`
- `always_with_diacritics`

## `capitalization_notes`

Краткие заметки о прописных/строчных:

- `lowercase_as_adjectival`
- `keep_capitalized`
- `contextual`

## `variants_found`

Найденные варианты в корпусе.

## `preferred_source`

Что стало основанием выбора:

- `our_books`
- `vedabase`
- `bbt`
- `bvks`
- `editor_decision`

## `source_examples`

Краткие ссылки/примеры из корпуса.

## `editor_decision`

Имя редактора / пометка утверждения.

## `status`

- `approved`
- `needs_review`
- `conflict`
- `draft`

## `notes`

Свободное поле для пояснений.

## 6. Правила наполнения

## 6.1. Источники

Использовать:

- локальное зеркало `vedabase`;
- книги Бхакти Викаши Свами;
- ваши текущие утвержденные книги.

## 6.2. Приоритеты

Если есть конфликт:

1. утвержденные книги издательства;
2. затем `vedabase`;
3. затем прецеденты БВКС;
4. затем решение редактора.

## 6.3. Что не делать автоматически

Не утверждать автоматически:

- спорное склонение;
- курсивность в неоднозначных случаях;
- расхождения между BBT и вашими книгами.

Такие случаи получают `status=needs_review` или `status=conflict`.

---

## 7. Pipeline сборки глоссария

## Этап 1. Extraction

Из корпуса автоматически извлекаются кандидаты:

- имена;
- термины;
- названия;
- курсивные фрагменты;
- формы с диакритикой;
- повторяющиеся слова с нетипичной орфографией.

## Этап 2. Normalization

Кандидаты группируются:

- по лемме;
- по близким вариантам;
- по контексту.

## Этап 3. Conflict detection

Выявляются:

- разные написания;
- разные склонения;
- разные решения по курсиву;
- разные решения по диакритике;
- разные решения по прописным/строчным;
- конфликт `vedabase` vs ваша книга.

## Этап 4. Editor review

Редактор/корректор утверждает спорные решения.

## Этап 5. Freeze for project

Для конкретной книги или серии книг фиксируется рабочая версия глоссария.

---

## 8. Что считать полезным выходом первой версии

Первая версия глоссария считается полезной, если она уже помогает:

- унифицировать написание;
- ловить расхождения;
- подсказывать склонение;
- подсказывать курсив;
- быстро отвечать, какая форма считается текущей нормой.
